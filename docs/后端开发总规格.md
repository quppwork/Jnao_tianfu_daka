## 二、产品目标

### 2.1 平台定位

面向 K12 的 **天赋成长平台**，四大模块共用后端与用户体系：

```
天赋测试 → 今日训练 → 知识答题 → 成长里程碑
```

**闭环（最终）：** 测天赋 → 定等级 → 每日 AI 训练方案 → 逐项打卡 → AI 答疑 → 成长追踪

### 2.2 MVP 与完整版

| 维度 | MVP（Phase 1，先上线） | 完整版（Phase 2–4） |
|------|----------------------|-------------------|
| 训练内容 | 每天 **1 条音频**（脑力奥秘 41 条） | **N 项顺序链**（音频+视频+指导） |
| 每日报告 | 可无 / 固定模板 | **AI 生成极简指令** |
| 打卡 | 点击即完成 | **逐项打卡，不可攒到最后** |
| 用户 | 孩子账号 + 家长手机号 | 手机号+微信登录、家长多孩 |
| 资源 | 阿里云 OSS **公开 URL** | 同左 + 资源库扩展至 162 条 |
| 审核 | 无 | 清北班导师审核 |

**原则：** 表结构与 API 按完整版设计，**实现按 Phase 分期交付**，避免推倒重来。

## 三、已确认业务规则（老板清单）

### 3.1 MVP 范围

| # | 规则 | 后端影响 |
|---|------|----------|
| 1 | MVP 只做音频，不做视频 | Phase 1 `video_url` 可空 |
| 2 | 以本地 **41 条 MP3** 为准 | `content_item` 导入 JSON |
| 3 | 家长注册为孩子开户；**当前只做孩子端** | `child_user.parent_phone` |
| 4 | 登录：手机号 + 微信（后期） | Phase 1 可简化认证 |
| 6 | **每天 1 条**，按 `lesson_sort` | 推送序号逻辑 |
| 7 | 8～9 条播完后 **循环** | `lesson_sort` 取模或回绕 |
| 8 | **必须先完成儿童天赋测评** | `GET /training/today` 无测评 → 403 |
| 9 | 可重测（后期可关） | `talent_assessment` 保留历史 |
| 10 | 完成训练 = **点击打卡即可** | 不校验播放时长 |
| 12 | 今日完成打卡 → **明日**推新内容；未完成 → 明日 **续推同一条** | 核心推送算法 |
| 14 | 使用 **儿童卷** `type=1` | JNAO 提交参数 |
| 16 | testresult 只走接口，失败直接报错 | 不做本地报告回退 |
| 17 | 注册：孩子姓名 + 家长手机号；其余进 App 后完善 | `profile_json` |
| 20 | 原 `video.jnao.com` URL 不可用 | **改用阿里云 OSS 公开 URL** |

### 3.2 AI 训练方案（老板口述）

```
每日开始 → 生成今日训练报告（极简指令）
         → 学员逐项完成（顺序执行，不可跳）
         → 每完成一项立即打卡（不能攒到最后）
         → 打卡数据 → 作为下次训练的承接基础
```

| 机制 | 说明 | Phase |
|------|------|-------|
| 顺序执行 | 做完一项 → 打卡 → 解锁下一项 | 2 |
| 学习连续性 | 今日方案基于昨日训练推演 | 2–3 |
| 每日训练报告 | 训练开始前生成，越少越好 | 3 |
| 智能时长推荐 | 今日计划时长 ≠ 昨日 → AI 询问多出的时间练什么 | 3 |
| 清北班审核 | 打卡后导师审核通过才解锁下一项 | 4 |
| 非清北班 | 打卡直接通过 | 1–3 |

### 3.3 待老板确认（开发 Phase 2+ 前）

1. 每日训练报告：**老板配课表** vs **AI 自动生成**？
2. 训练项顺序/内容：**预设模板** vs **AI 动态安排**？
3. **12 能力类型**是否保留？
4. 导师审核在 **App 内**还是外部？
5. 每日计划训练时长：**学员填 / 老板预设 / 系统按等级**？

## 七、核心算法

### 7.1 每日内容推送（Phase 1，必做）

```text
输入: child_user_id, plan_date（通常今天）

1. 查最新 talent_assessment
   └─ 无 → 返回 403「请先完成天赋测评」

2. 查昨日 daily_plan / training_plan + 打卡状态
   ├─ 昨日已完成打卡 → progress_index += 1（下一条 lesson_sort）
   └─ 昨日未完成/未打卡 → progress_index 不变（续推同一条）

3. 从 content_item 按 talent_code + lesson_sort 取第 N 条
   ├─ 跳过 lesson_sort=0 的「我是冠军」（可选）
   └─ 超出系列条数 → 循环回第一条（老板 #7）

4. 写入/返回今日 plan（Phase 1: N=1 项）
```

**进度计算：** 可用 `COUNT(checkin_record)` 或单独 `user_progress` 字段记录当前 `lesson_sort` 序号。

### 7.2 顺序解锁（Phase 2）

```text
training_item[1..N]
  item[i].status = locked | active | done
  item[1] 默认 active
  item[i] 打卡完成 + (清北班: review approved) → item[i+1] = active
  禁止跳过：item[i] 未完成时 item[j>i] 不可打卡
```

### 7.3 学习连续性（Phase 2–3）

```text
生成今日方案时 input:
  - 昨日最后一项完成情况
  - 昨日 report_text 摘要
  - 历史薄弱 ability_type（可选）
  - 学员 talent_code + training_level
```

### 7.4 AI 每日报告（Phase 3）

```text
TrainingPlanGenerator.generate():
  1. 收集承接上下文（7.3）
  2. 编排 training_item 列表（资源库 / 规则模板）
  3. 调豆包：system=「生成极简今日指令，2-4 句」
  4. 写入 training_plan.report_text
```

### 7.5 智能时长推荐（Phase 3）

```text
if today.planned_minutes > yesterday.planned_minutes:
    delta = today - yesterday
    → 返回追问：「多出 {delta} 分钟，想重点练什么？」
    → 用户有回复 → 按指定重排 item
    → 无回复 → AI 推荐薄弱项
```

### 7.6 清北班审核（Phase 4）

```text
打卡 → training_record.review_status = pending
导师 POST /api/training/review/{id} → approved | rejected
approved → 解锁下一 training_item
```
